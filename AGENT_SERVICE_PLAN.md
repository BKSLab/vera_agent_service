## 0. Назначение и рамки сервиса

Agent Service — оркестратор диалога с Верой. Единственный сервис в трёхсервисной архитектуре (`AGENT_VERA_ARCHITECTURE.md`, раздел "Три сервиса"), который знает о сессии пользователя, истории сообщений и решает, нужен ли инструмент для ответа. Не хранит контент базы знаний (RAG Service) и не знает деталей реализации инструментов (MCP Tools Server) — общается с ними только через внешние контракты (раздел 3).

**В рамках этого плана:**
- Consumer очереди `agent.requests` (RabbitMQ) — асинхронный вход задачи от Next.js Proxy.
- Граф LangGraph: `analyze_intent → call_kb_search → generate_with_context / generate_direct`.
- Клиент к MCP Tools Server (`langchain-mcp-adapters`) — против конфигурируемого `MCP_SERVER_URL`, с моком на время отсутствия реального сервера.
- Redis checkpointer LangGraph — состояние диалога между сообщениями сессии.
- SSE-доставка ответа клиенту.
- Системный промпт, `AgentState`.
- Наблюдаемость: OpenTelemetry/openinference → Arize Phoenix.
- Docker-образ, тесты, документация контрактов для смежных команд.

**Вне рамок этого плана:**
- Сам **MCP Tools Server** — отдельный репозиторий/трек, следующий шаг после этого плана. Здесь фиксируется только ожидаемый контракт тула `kb_search` (раздел 3.3), под который MCP-сервер должен подойти.
- **RAG Service** — уже реализован (`vera_rag_service`, production-ready).
- **Next.js Proxy** и фронтенд-виджет чата — реализуются командой Frontend по контракту из раздела 3.1/3.2.
- Инструменты итерации 2 и их источник данных (`api_work_for_everyone`).
- Развёртывание самого RabbitMQ/Redis/Phoenix как общей инфраструктуры проекта (здесь — только `docker-compose.yml` для локальной разработки и конфигурация подключения).

**Явное следствие отсутствия MCP Tools Server:** узел `call_kb_search` (этап 4/5) реализуется и тестируется против мока MCP-клиента. Реальный end-to-end прогон `Agent → MCP → RAG` невозможен в рамках этого плана и помечается заблокированным до появления MCP Tools Server — см. раздел 4.

## 0.1. Зафиксированные технические решения

| Решение | Выбор | Причина |
|---|---|---|
| RabbitMQ-клиент | `aio-pika` напрямую, тонкий consumer-модуль без брокерского фреймворка поверх | Соответствует выбору "LangGraph, не PydanticAI" в `AGENT_VERA_ARCHITECTURE.md` — явный код важнее магии фреймворка |
| Redis checkpointer | Пакет `langgraph-checkpoint-redis` (`AsyncRedisSaver`) | Официальная интеграция LangGraph с Redis; Redis как checkpointer зафиксирован в WBS (`AGENT_VERA_WBS.txt`, п. 2.8) |
| LLM-интеграция в графе | LangChain-совместимая chat-модель (`langchain-openai.ChatOpenAI`, указывает на OpenAI-совместимый эндпоинт из `.env`), не сырой `LlmClient` из `LLM_CLIENT_REFERENCE.md` | Нужны нативные стриминг (`.astream()`) и tool-calling (`.bind_tools()`) — на них построен LangGraph. `LlmClient` из `LLM_CLIENT_REFERENCE.md` не умеет читать потоковые SSE-чанки от провайдера (см. этап 3) |
| SSE-доставка ответа | Agent Service сам держит `GET /sse/{session_id}`, токены — из per-session `asyncio.Queue` в памяти процесса, которую наполняет RabbitMQ-consumer в момент стриминга графа | Работает "из коробки" при одном инстансе (следующая строка); не требует Redis Pub/Sub на старте |
| Количество реплик Agent Service | Один инстанс на итерацию 1 | Необходимое условие для in-process SSE-очереди выше; такое же допущение уже сделано для RAG Service (`vera_rag_service/entrypoint.sh`, ARCH-9/10). При масштабировании — переход SSE-очереди на Redis Pub/Sub, отдельная будущая задача |
| Persistent-хранилище (Postgres) | Не используется в MVP | У Agent Service нет доменных данных для хранения — состояние диалога живёт в Redis (TTL), логи запросов — в structured logging + Phoenix. Открытый вопрос — см. раздел 4 |
| MCP-клиент на время отсутствия MCP Tools Server | `langchain-mcp-adapters.MultiServerMCPClient` собирается по-настоящему, но в тестах и локальной разработке подменяется мок-HTTP/SSE-сервером с одним тулом `kb_search`, отдающим фикстуры | MCP Tools Server — отдельный трек (решение из обсуждения концепции, 2026-07-09) |
| Число LLM-вызовов на реплику пользователя | Фиксированно **2**: (1) `analyze_intent` — короткий вызов **без стриминга**, только решение "нужен ли `kb_search`" + его аргументы (tool-calling/structured output); (2) `generate_with_context`/`generate_direct` — отдельный **стримингованный** вызов с финальным ответом | Слияние классификации и генерации в один стримингованный вызов (с ранней детекцией `tool_calls` в потоке) — поведение, зависящее от конкретного провайдера; рискованно при заявленном "провайдер конфигурируется" (`AGENT_VERA_ARCHITECTURE.md`). Два простых вызова предсказуемее в реализации и тестировании. TTFT-бюджет (≤5с медиана, `AGENT_VERA_ACCEPTANCE_CRITERIA.html`, раздел 3) держать в уме на этапе 4 — при нехватке запаса оптимизация до одного вызова возможна отдельной задачей позже, не блокирует итерацию 1 |
| Обработка ошибки MCP/RAG внутри графа | **Не исключение наружу графа.** Узел `call_kb_search` перехватывает ошибку/таймаут MCP-клиента (после исчерпания его собственных повторов, этап 3) и кладёт в `AgentState` явный признак "поиск недоступен"; `generate_with_context` в этом случае формирует честный ответ-извинение, а не падает и не выдумывает факты | Ошибка стороннего сервиса — ожидаемая на практике ситуация (MCP Tools Server ещё не существует, раздел 0), она не должна приводить к падению обработки всего сообщения и попадать в RabbitMQ-retry (см. следующую строку) |
| RabbitMQ retry/DLQ — область применения | Только **системные** сбои consumer'а (невалидный payload, необработанное исключение **до начала стриминга** ответа). Доменные сбои внутри графа (MCP/LLM недоступны) — не повод для requeue, они обработаны деградацией (строка выше) | Requeue всего сообщения после того, как часть токенов уже ушла клиенту через SSE, создаёт дублирование/рассинхронизацию потока. Retry на уровне сообщения оправдан только пока ничего ещё не отправлено пользователю |
| `history` в payload RabbitMQ | **Убран из контракта.** Единственный источник истории диалога — Redis-checkpointer LangGraph, ключ треда — `session_id` (этап 5) | Дублирование источника правды не нужно и создаёт риск рассинхронизации; чекпоинтер и так восстанавливает историю по `session_id` |
| `AgentState.messages` — способ накопления истории | `Annotated[list[BaseMessage], add_messages]` (LangGraph reducer), не голый `list[BaseMessage]` | Без reducer'а новое входящее сообщение **затирает** историю, восстановленную checkpointer'ом, вместо того чтобы дополнить её — это единственный механизм, которым в графе реально работает решение "убрать `history` из payload" (строка выше) |
| SSE-канал на сессию | Один активный `session_id` = один активный `asyncio.Queue`; повторное подключение (например, вторая вкладка) замещает предыдущую регистрацию | Явное упрощение для итерации 1, согласуется с допущением "один инстанс" (строка выше). Несколько одновременных вкладок на одну сессию — вне рамок итерации 1 |

Он никогда не сбрасывался обратно после штатной остановки одного тестового сервера, из-за чего **следующий** поднятый в том же процессе мок-сервер немедленно обрывал свои SSE-соединения — при запуске тестовых файлов по отдельности незаметно, проявлялось только при полном прогоне `pytest tests/`. Исправлено сбросом `AppStatus.should_exit = False` в `finally`-блоке `run_mock_mcp_server` (правка в артефакте Этапа 3, `tests/fixtures/mock_mcp_server.py`). Полный прогон `pytest tests/` теперь стабильно зелёный при повторных запусках (проверено дважды подряд) — до этой находки 6–8 из 55 тестов случайно падали именно при полном прогоне, хотя каждый файл по отдельности был зелёным.

Итого 55/55 тестов проекта зелёные (стабильно, дважды подряд полным прогоном), `ruff check .` чист.

### Этап 6 — RabbitMQ consumer (`agent.requests`) ⏳ Запланировано

- [ ] 6.1 `app/messaging/schemas.py` — Pydantic-схема входящего сообщения (см. контракт, раздел 3.1); ограничение максимальной длины `message` (конкретное число символов зафиксировать при реализации — защита от аномально больших сообщений, см. раздел 6 "Открытые вопросы")
- [ ] 6.2 `app/messaging/consumer.py` — `aio-pika`-consumer очереди `agent.requests`, `prefetch_count=1` (один инстанс обрабатывает одно сообщение за раз — согласуется с допущением "один инстанс", раздел 0.1, и с in-process SSE-очередью, этап 7): разбор payload → вызов графа (стриминг через `graph.astream_events`/`astream(stream_mode="messages")`) → передача чанков в SSE (этап 7) → `ack` сообщения

Сбои **внутри графа** (MCP/LLM недоступны) не относятся к этой политике — они обработаны деградацией на уровне графа (этап 4.2/4.3, раздел 0.1) и заканчиваются нормальным `ack`, не requeue
- [ ] 6.4 `app/exceptions/messaging.py`: невалидный payload → в DLQ по политике 6.3; исключение, возникшее **после того как стриминг ответа клиенту уже начался**, — `ack` сообщения (без requeue, см. раздел 0.1) + SSE `error`-событие клиенту (этап 7.2)
- [ ] 6.5 Юнит-тесты на моке `aio-pika`-соединения: (а) невалидный payload → DLQ после 3 попыток; (б) исключение до старта стриминга → requeue согласно политике; (в) исключение после старта стриминга (например MCP падает не в самом узле 4.2, а по дороге к нему в неожиданном месте) → `ack` + SSE-ошибка, без requeue и без повторной обработки того же сообщения
- [ ] 6.6 Интеграционный тест на реальном RabbitMQ (`docker-compose.yml`, сервис `rabbitmq`): публикация тестового сообщения → сообщение доходит до графа (граф — на моках LLM/MCP)

**Definition of Done:** сообщение, опубликованное в `agent.requests`, обрабатывается и в конце вызывает потоковую передачу в SSE-очередь сессии (этап 7); намеренно невалидное сообщение уходит в `agent.requests.dlq` после 3 попыток; сбой внутри графа **после** начала стриминга не приводит к повторной обработке того же сообщения (нет дублирования токенов в SSE).

## 3. Контракты (раздел для смежных команд)

### 3.1 Frontend / Next.js Proxy → RabbitMQ

Очередь `agent.requests`, payload (Pydantic-схема — `app/messaging/schemas.py`, этап 6.1):

```json
{
  "session_id": "uuid",
  "user_id": null,
  "message": "Какая квота на трудоустройство инвалидов?"
}
```

`user_id` передаётся уже провалидированным Next.js Proxy (Agent Service не проверяет JWT самостоятельно — доверенная граница на стороне прокси). Поле `history` **сознательно отсутствует** в контракте (решение — раздел 0.1): единственный источник истории диалога — Redis-checkpointer LangGraph, ключ треда — `session_id` (этап 5, работает через reducer `add_messages`, этап 1.1); фронтенду не нужно ни хранить, ни пересылать историю для работы агента.

Retry: 3 попытки, экспоненциальный backoff, dead-letter queue `agent.requests.dlq` — только для системных сбоев обработки сообщения (раздел 0.1, этап 6.3), без ручного разбора сообщение не переигрывается.

### 3.2 Agent Service → Клиент (SSE)

`GET /sse/{session_id}` (эндпоинт Agent Service, доступность через Nginx — вне рамок этого плана). Одно активное соединение на `session_id` (раздел 0.1) — повторное подключение (например, вторая вкладка) замещает предыдущее; несколько одновременных вкладок на одну сессию не поддерживаются в итерации 1.

События (`app/streaming/sse.py`, этап 7.2):

```
data: {"type": "token", "content": "Квота"}
data: {"type": "token", "content": " на трудоустройство"}
...
data: {"type": "done"}
```

Ошибка (в том числе — исчерпание retry в RabbitMQ, недоступность MCP Tools Server):

```
data: {"type": "error", "detail": "человекочитаемое сообщение"}
```

## 6. Открытые вопросы к моменту начала работы

Список решений, которые лучше подтвердить до или в начале соответствующего этапа, а не по ходу дела. Часть вопросов из первой версии этого плана решена по итогам ревью (число LLM-вызовов, поле `history`, поведение SSE при позднем подключении, retry только для системных сбоев — см. раздел 0.1) — здесь остались только вопросы, где предложено значение по умолчанию, но нет формального подтверждения:

3. Конкретный лимит длины `message` в payload RabbitMQ (этап 6.1).
4. Конкретные значения таймаута и числа повторов на уровне MCP-клиента (этап 3.2) — независимо от retry-политики самого RabbitMQ-сообщения (этап 6.3).
